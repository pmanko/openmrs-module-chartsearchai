# Honest drift metric

An offline evaluation harness that scores chartsearchai answer quality (drift, recall,
abstention) against a **human-adjudicated gold standard of record UUIDs** — not against
model-derived keyword lists.

## Why this exists

The previous ad-hoc scorer (`score_eval.py`) flagged "drift" using a hardcoded list of
off-topic terms derived from one model's (Gemma-4-E2B) known mistakes. That list is blind to
*novel* drift from any other model, so it produces misleading verdicts. The motivating case:

| metric | E2B | E4B | reading |
|---|---|---|---|
| `score_eval` (keyword, E2B-tuned) | 6 drift | **1 drift** | "E4B is a big win — ship it" |
| **this metric — mean F1** | **0.538** | 0.504 | E4B slightly worse |
| **this metric — abstention acc** | **0.82** (9/11) | 0.73 (8/11) | E4B worse |
| **this metric — off-topic citations** | **76** | 82 | E4B worse |

The keyword scorer said E4B was a clear improvement; measured honestly, E4B is *slightly worse
on every axis* — it relocates failures (fixes betty|eye, mark|allergies; breaks karen|kidney,
richard|mental; invents two new "programs → appointments/visits" abstention failures; explodes
mark|heart to 54 off-topic citations). A keep-if-better automation pointed at `score_eval`
would have shipped a regression. This metric refuses it. **It is honest by construction: it
compares cited UUIDs to a human gold set, so it cannot inherit a single model's blind spot.**

## Eval set

4 OpenMRS reference-demo patients × 8 scope queries = 32 cells (synthetic demo data):

- betty `4acc0b80-83c4-40f7-86fd-0e11a68dd405`, richard `07e26b8e-00a9-4b31-b805-3560ad4e9e2e`,
  karen `be83f269-66bd-4ba1-80ec-cc62d0d0c84e`, mark `61d0a9db-d35f-40c9-aeae-ccd264470de5`
- queries: programs, allergies, medications, eye, heart, fractures, kidney, mental
  (see `capture_eval.sh`)

## Files

- `capture_eval.sh <outdir>` — fires the 32 cells at the live REST endpoint, saving the full
  response JSON (references carry `resourceUuid`) per cell. Requires a running standalone with
  `chartsearchai.querystore.enabled=true`, `embedding.preFilter=true`. Override `OPENMRS_AUTH`
  (default `admin:Admin123`) and `OPENMRS_REST` for a non-default standalone (also used by
  `resolve_unknowns.py`).
- `metric_gold.json` — per cell: `present` (is the topic actually in the chart), `ontopic`
  (UUID→text of genuinely on-topic records), `focus_uuids` (the top-30 the search returned).
- `offtopic_adj.json` — out-of-focus cited records adjudicated on/off-topic (`_ontopic` map for
  on-topic, per-cell lists for off-topic). The gold *expands* here as new models cite new
  records.
- `focus_labels.json` — the seed adjudication (on-topic *ranks* within each top-30 focus set)
  that `build_gold.py` turns into `metric_gold.json`.
- `build_gold.py` — focus labels (+ focus dumps) → `metric_gold.json`.
- `resolve_unknowns.py <capture_dir>` — lists cited records not yet in gold and resolves their
  names via OpenMRS REST (`/obs`, `/condition`, `/patientdiagnoses`, …) so they can be adjudicated.
- `metric_score.py <capture_dir> [offtopic_adj.json] [metric_gold.json]` — UUID set comparison → per-cell
  precision/recall/F1 (present cells) + abstention correctness (absent cells), aggregate.

## Workflow to score a new model

1. Point the standalone at the model, then `capture_eval.sh /tmp/eval_capture/<MODEL>`.
2. `resolve_unknowns.py /tmp/eval_capture/<MODEL>` → adjudicate any new out-of-focus citations,
   merge into `offtopic_adj.json` (on-topic into `_ontopic`, else the per-cell off list).
3. `metric_score.py /tmp/eval_capture/<MODEL> offtopic_adj.json`.

`metric_gold.json` requires no focus dumps to *score* — dumps are only needed to (re)build it
via `build_gold.py`. The focus dumps live outside this dir (regenerated by flag-gated
instrumentation; not committed).

## Limitations (v1)

- **Recall** is measured over the focus-30-anchored on-topic set (+ adjudicated out-of-focus
  on-topic records), so it does not penalize on-topic records that *no* evaluated model has
  cited *and* that sit outside the search's top-30 — a retrieval-recall concern, separate from
  the LLM drift this targets. The gold expands as more models are scored.
- The on/off-topic standard embeds judgment calls — notably **raw BP/pulse/SpO2 vitals are
  off-topic for "heart problems"** (a vital reading is not a cardiac condition). This drives the
  large heart-cell drift counts; it is explicit and adjustable (edit the gold, re-score). F1 and
  abstention are far less sensitive to it than the raw drift count.
- Gold is fixed to these 32 cells; new patients/queries need fresh adjudication.

## Measured results on the standalone gold (2026-06-12)

Baseline (Gemma 4 E2B, full-chart, `metric_gold.standalone.json`): **meanF1 0.464 ·
abstention 1.00 (11/11) · off-topic citations 41** — reproduced bit-identically across two
captures (zero run-to-run noise on this protocol; greedy decode + warm prompt cache).

Two reasoning-length experiments were run through this gate with thresholds locked in
advance (no regression on any axis; ≥15% output-token reduction to be worth shipping).
Both failed, through opposite mechanisms:

| arm | meanF1 | abstention | drift | out tokens | verdict |
|---|---|---|---|---|---|
| baseline | 0.464 | 1.00 | 41 | 6,448 | — |
| grammar truncation (`reasoningMaxChars=400`, PR #45) | 0.428 | 0.91 | 47 | 6,408 | FAIL: cut mid-derivation → unsafe citations appear (incl. a false citation on an absent cell); no net token saving |
| prompt-shaped brevity ("at most two short sentences" in the system prompt) | 0.410 | 1.00 | 19 | 3,957 | FAIL: recall collapse — clean but INCOMPLETE answers (drift improves, on-topic records go missing), violating the complete-enumeration safety property |

Conclusion: E2B needs its full reasoning scratchpad to enumerate completely. Shortening it
— by force or by instruction — trades one safety property for another. Do not revisit
without a different model or a mechanism that preserves completeness.
